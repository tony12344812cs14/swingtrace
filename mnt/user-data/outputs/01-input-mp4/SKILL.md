# 模組一:輸入處理(MP4)

## 概覽

這個模組的唯一職責:**把一支來路不明的 MP4,變成乾淨、標準化、帶完整中繼資料的幀序列**,並在源頭擋掉不能分析的影片。後面模型測得準不準,有一半取決於這裡。

定位是**賽後離線分析**,所以不追求即時,可以多花時間做檢查與標準化。

## 快速參考

| 任務 | 做法 |
|------|------|
| 抽幀 | ffmpeg 解碼成圖片序列或逐幀讀取 |
| 讀 metadata | 用 ffprobe 取得 fps、解析度、總幀數、時長 |
| 合格檢查 | fps 門檻、畫面穩定度、球場線可見性 |
| 輸出 | 幀序列 + `meta.json`(存檔,別只放記憶體) |

## 為什麼 fps 是這個模組的命根子

下游的**速度估算完全靠 fps 回推**,fps 抓錯,速度就全錯。所以:

- 一定用 ffprobe 讀**實際 fps**,不要假設 30。手機高幀率模式常是 120/240。
- 注意**可變幀率(VFR)**:有些手機影片不是固定幀率,逐幀的時間間隔不一致。若是 VFR,要嘛轉成固定幀率(CFR),要嘛在 metadata 記下每幀的真實時間戳,別假設等間距。
- fps 太低(例如 30fps 拍網球發球或大力抽擊)→ 揮拍那隻手臂在相鄰幀之間移動太大、動態模糊嚴重,手腕/拍頭的軌跡會稀疏又抖。發球的揮拍是全片最快的動作,對 fps 最敏感。這種要**警告甚至退件**,而不是硬跑出一個錯的結果。

```bash
# 讀真實 fps / 解析度 / 幀數(務必用實際值,不要猜)
ffprobe -v error -select_streams v:0 \
  -show_entries stream=r_frame_rate,avg_frame_rate,width,height,nb_frames \
  -of json input.mp4
```

## 合格檢查(上傳即擋,別讓爛影片進管線)

我們先前的共識是「先狠狠約束輸入」。這個模組就是執行那道閘門:

1. **fps 門檻**:低於某個值(快速揮拍要求更高)→ 警告。
2. **畫面穩定度**:預期是**固定機位(腳架)**。可用相鄰幀的全域位移估計晃動程度,超標 → 警告(手持影片會讓姿態估計與軌跡抽取變得不穩)。
3. **球員入鏡完整性**:姿態估計需要看得到揮拍者的上半身(至少肩—肘—腕這條揮拍鏈)。人被裁切、背對鏡頭、揮拍手臂長時間被遮擋 → 標記。網球**發球**時手臂會繞到頭後(trophy / 撓背位置)造成短暫自我遮擋,這是正常的、靠模組三補,但若整段都看不到揮拍側就要退件。
4. **球員夠大夠清楚**:拍太遠導致人只佔畫面一小塊,關節定位會不準 → 提示靠近或拉近。
5. **單人 / 多人**:畫面若有多人,要能鎖定要分析的那一位(見模組二),否則提示。

檢查不過的處理原則:**明確告知原因 + 給拍攝建議**(固定機位、揮拍者完整且夠大入鏡、揮拍側盡量不被遮擋、高 fps、避免逆光),而不是默默產出爛結果。

## 輸出契約(下游模組依賴這個)

固定輸出格式,讓模型模組不用管影片怎麼來的:

- **幀序列**:標準化後的逐幀影像(統一色彩空間;若做了去抖動要註明)。
- **`meta.json`**,至少包含:
  - `fps_real`(實際幀率)、是否 VFR、每幀時間戳(VFR 時必要)
  - `width` / `height` / `total_frames` / `duration`
  - `stroke_type`(擊球類型:正手 / 單手反拍 / 雙手反拍 / 發球 / 截擊 — 影響取哪些關節與揮拍階段判讀,可由使用者標或後續自動分類)
  - `dominant_hand`(慣用手:左/右 — 決定取哪一側的肩肘腕,見模組二)
  - `two_handed_backhand`(是否雙手反拍 — 若是,反拍時非慣用手也在拍上,模組二要一併取兩側手腕)
  - `checks`(穩定度、fps、球員入鏡完整性的檢查結果)
  - `source_hash`(原始檔雜湊,確保可重現)

## 雷區與提醒

- **不要假設等間距時間**:VFR 影片用幀號 × (1/fps) 算時間會錯,速度跟著錯。
- **中繼結果一定落地**:抽好的幀和 meta 存成檔,下游模組改邏輯時不用重新解碼整支影片。這在開發期省你大量時間。
- **去抖動是雙面刃**:穩定化會改動像素座標,而揮拍軌跡就是手腕/拍頭的像素座標,務必讓姿態估計與軌跡抽取吃的是同一份(穩定化後的)畫面,否則軌跡會被相機運動污染。
- **慣用手與擊球類型在這裡定案**:左手 / 右手、是否雙手反拍、是正手/反拍/發球,都會影響後兩個模組怎麼取關節與判讀動作。慣用手與雙手反拍務必在輸入階段就確定、寫進 meta;擊球類型可先讓使用者標,或留給後續自動分類。
