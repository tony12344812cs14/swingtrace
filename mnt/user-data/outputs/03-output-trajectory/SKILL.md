# 模組三:軌跡輸出

## 概覽

職責:把模型模組吐出的**一堆逐幀手腕/拍頭候選點**,變成**一條乾淨、連續、動作上合理的揮拍軌跡**,再疊回影片給人看,並算出揮拍速度等衍生數據。這是把「偵測」變成「結果」的最後一哩。

因為是**離線分析**,可以同時看前後幀(雙向),這對修補快速揮拍時漂掉/遮擋的點是巨大優勢,務必吃滿。

## 快速參考

| 任務 | 做法 |
|------|------|
| 連點成線 | 軌跡關聯,把逐幀手腕/拍頭點串成連續路徑 |
| 補斷點 | 雙向追蹤 + 動作連續性,反推遮擋/低信心的缺失點 |
| 平滑 | 整段最佳化(平滑但保留接觸瞬間的轉折) |
| 速度 | 用實際 fps + 尺度參照換算成真實速度 |
| 輸出 | 疊軌跡影片 + 軌跡資料(JSON) |

## 善用離線雙向優勢

即時版只能看「過去到現在」,離線版前後幀都有。具體利用:

- **雙向追蹤與軌跡修補**:手腕/拍頭被身體或球拍遮擋、或快速揮拍時信心過低的那幾幀,用前後段軌跡**反推補回來**。
- **整段最佳化**:不要逐幀貪心地連最近的點(容易被雜訊或跳人帶歪),把整條軌跡當成一個最佳化問題,用平滑度與動作連續性當約束,一次解出最可能路徑。
- **多次 pass**:先粗連 → 再修斷點 → 再平滑,分階段好除錯。

## ⚠️ 最重要的雷:揮拍軌跡不是彈道,別套球的物理

球的飛行(拋物線、受空氣阻力與旋轉影響)那套**完全不適用**於揮拍軌跡——你畫的是手腕/拍頭的路徑,受**人體手臂運動學**支配,不是空中飛行:

- 揮拍軌跡是一條**由肩—肘—腕關節帶動的平滑弧線**(引拍→揮出→接觸→隨揮),沒有空氣彈道可言。平滑時用**運動連續性 / 樣條曲線**這類資料驅動的方式,別套任何拋物線或重力模型。
- **接觸瞬間是關鍵轉折,不能被平滑抹掉**:揮拍在擊球點附近方向/速度急轉,過度平滑會把這個最有分析價值的點磨平。平滑要「整體順、轉折保」。
- **擊球類型影響的是判讀基準,不是平滑公式**:網球的正手、單/雙手反拍、發球、截擊,軌跡形狀與幅度差很多(讀 metadata 的 `stroke_type`),但全都是人體弧線,平滑邏輯共用即可,差別在後續怎麼解讀,而非怎麼補點。發球是大幅過頭弧線、截擊幾乎沒有引拍,判讀時要分開看。

## 速度估算

- 用**實際 fps**(來自模組一的 metadata,注意 VFR)回推手腕/拍頭的幀間位移 → 揮拍速度。
- 像素位移要先換算成真實距離才是真速度。揮拍場景沒有球場 homography 可用,改用**尺度參照**:畫面中已知長度的東西(球拍長度、球員身高、肩寬等)當比例尺,把像素轉成公尺/秒或公里/時。沒有可靠參照時,只能給「相對/未校正」速度並註明。
- 注意這是 **2D 投影速度**:揮拍平面斜對鏡頭時會被低估,要連同視角限制一起標註(見模組二)。
- fps 太低時速度誤差大,要在輸出標註可信度,別給一個看似精確其實亂猜的數字。

## 輸出契約

- **疊軌跡影片**:把連續揮拍軌跡(手腕或拍頭)畫回原影片(可保留淡出尾巴看出方向),也可一併疊上手臂骨架。這是 MVP 的主要交付物。
- **軌跡資料(JSON)**:逐點的幀號、座標(手腕/拍頭,標明哪一種)、速度、是否為修補點/外推點。讓結果可被後續分析(揮拍階段、揮拍平面、動作範圍)再利用。
- **中繼結果落地**:軌跡序列存檔,改視覺化或下游分析時不必重跑前面。

## 與後續功能的接口(先留好,不一定現在做)

這個模組的輸出是很多揮拍分析的輸入,設計時把欄位留好:

- 軌跡的速度峰值 / 方向急轉 → **接觸瞬間**偵測。
- 軌跡分段 → **揮拍階段**切分(引拍、向前揮、接觸、隨揮;發球另有拋球、撓背、加速、接觸的特殊結構)。
- 整條弧線的形狀與起訖 → **揮拍平面、動作範圍(ROM)**。
- 軌跡形狀特徵 → **擊球類型分類**(正手 / 單手反拍 / 雙手反拍 / 發球 / 截擊),可回填 `stroke_type`。
- 多次同類揮拍疊圖 → **一致性 / 穩定度**分析(同一動作每次像不像)。

## 雷區與提醒

- **分清楚「觀測點」「修補點」「外推點」**:補出來的、以及拍頭外推的點都要標記,輸出可用不同樣式呈現,也方便評估實際偵測率。
- **平滑過頭會說謊**:過度平滑會把接觸瞬間的急轉抹平,而那正是揮拍分析最有價值的點。平滑強度要保住關鍵轉折。
- **沒尺度參照就別亂報真實速度**:找不到可靠比例尺時只能給畫面相對量;斜角視角的 2D 速度會低估,都要明確標註,別讓使用者誤以為是實測值。
